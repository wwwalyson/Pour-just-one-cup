# 机械臂上位机软件设计

## 第二部分 系统组成及功能说明

### 2.5 机械臂上位机软件设计（黄启超）

#### 2.5.1 机械臂上位机软件概述

本作品的机械臂控制部分以 LeArm 六自由度机械臂为执行机构，采用"上位机（PC）逆运动学解算 + 下位机（STM32）舵机驱动"的分布式控制架构。上位机运行 ROS2 节点完成目标坐标接收、逆运动学求解、关节空间轨迹规划和串口指令下发；下位机 STM32 负责串口协议解析、舵机总线通信和硬件驱动。两者通过 USB 转串口（USART1，波特率 9600）进行数据交互。

上位机软件基于 ROS2 Humble 框架开发，由 C++ 和 Python 混合编程实现，主要包含四个功能模块：串口通信桥接节点（arm_base）、逆运动学求解节点（ik_node）、坐标指令节点（arm_cmd）和任务调度节点（tea_task_node）。各模块通过 ROS2 Topic 解耦通信，既便于分模块独立调试，也便于后续替换机械臂硬件或升级运动学算法。

从系统关系上看，视觉模块（深度相机 + YOLO）提供目标空间坐标，手眼标定模块将相机坐标转换到机械臂基坐标系，语言模块给出任务意图，而机械臂上位机软件负责将所有上层指令转化为下位机可执行的舵机控制帧，是连接感知决策与物理执行的关键桥梁。

#### 2.5.2 上位机-下位机串口通信协议

上位机与 STM32 下位机之间通过 USB 转串口（USART1）进行通信，采用自定义二进制帧协议。帧格式如下：

```
+--------+--------+----------+-----+-------------------+
| 0x55   | 0x55   | data_len | cmd | payload           |
+--------+--------+----------+-----+-------------------+
  1 byte   1 byte   1 byte    1 byte   (data_len - 2) bytes
```

| 字段 | 长度 | 说明 |
|------|------|------|
| header_1 | 1 | 固定 0x55 |
| header_2 | 1 | 固定 0x55 |
| data_len | 1 | 总长度 = cmd(1) + 实际数据长度，最小值为 2 |
| cmd | 1 | 命令码 |
| buffer[] | N | 实际数据，共 data_len - 2 字节 |

上位机主要使用以下三种命令码与下位机交互：

| 命令码 | 名称 | 用途 | 数据方向 |
|--------|------|------|----------|
| CMD_VERSION_QUERY (1) | 版本查询 | 握手验证，确认串口通信正常 | 上位机→下位机→上位机 |
| CMD_COORDINATE_SET (4) | 增量坐标控制 | 发送 ΔX/ΔY/ΔZ 增量，由下位机闭源 IK 库解算 | 上位机→下位机 |
| CMD_MULT_SERVO_MOVE (3) | 多舵机直接控制 | 绕过下位机闭源 IK，直接发送 6 路舵机脉宽 | 上位机→下位机 |
| CMD_ANGLE_BACK_READING (13) | 舵机角度回读 | 读取 6 个舵机当前脉宽值，用于闭环验证 | 上位机→下位机→上位机 |

**（1）增量坐标控制帧（CMD_COORDINATE_SET = 4）**

该帧为传统控制模式，发送末端坐标增量，由 STM32 端闭源运动学库 LeArm.lib 完成逆运动学解算。帧结构如下：

```
55 55 05 04 ΔX ΔY ΔZ
│  │  │  │  │  │  └── Z 轴增量 (int8_t, 单位 0.1 cm)
│  │  │  │  │  └───── Y 轴增量 (int8_t)
│  │  │  │  └──────── X 轴增量 (int8_t)
│  │  │  └─────────── 命令码 = 4
│  │  └────────────── data_len = 5 (cmd 1字节 + 数据 3字节... 实际 1+3=4, len=5 是因为定义 data_len=2+payload_size)
│  └───────────────── 帧头2
└──────────────────── 帧头1
```

增量换算关系：坐标实际值(cm) = 发送值 / 10.0，即每个增量单位对应 0.1 cm 的位移。下位机默认起始坐标为 (X=15.0, Y=0.0, Z=2.0) cm，每个增量帧在默认坐标上累加。坐标范围受下位机固件限制：X∈[10.0, 20.0] cm，Y∈[-10.0, 10.0] cm，Z∈[0.0, 25.0] cm。

该模式的局限性在于：依赖 STM32 端闭源逆运动学库 LeArm.lib，上位机无法直接控制关节角度和运动时间；坐标采用增量累加方式，长期运行存在累积误差；运动轨迹完全由下位机决定，上位机无法进行轨迹规划。因此，系统新增了 IK 模式作为主要工作模式。

**（2）多舵机直接控制帧（CMD_MULT_SERVO_MOVE = 3）**

IK 模式下，上位机在 ROS2 侧自行完成逆运动学求解，将关节角度转换为 6 路舵机脉宽值，直接通过 CMD_MULT_SERVO_MOVE 帧下发。帧结构如下：

```
55 55 data_len 03 count time_l time_h [id duty_l duty_h]...
│  │  │        │  │     │      │      │
│  │  │        │  │     │      │      └── 每3字节一组 (舵机ID, 脉宽低字节, 脉宽高字节)
│  │  │        │  │     │      └── 运行时间高8位 (ms)
│  │  │        │  │     └── 运行时间低8位 (ms)
│  │  │        │  └── 舵机数量 (6)
│  │  │        └── 命令码 = 3
│  │  └── data_len = 5 + count×3  (例如 count=6 时 data_len=23)
│  └── 帧头
```

6 舵机对应的关节映射关系如下：

| 舵机 ID | 关节名称 | 关节功能 | 角度范围 | 脉宽范围 | 脉宽换算公式 |
|---------|----------|----------|----------|----------|-------------|
| 6 | shoulder_pan | 底座旋转 | -90° ~ 90° | 125 ~ 875 | pulse = 500 + 4.1667 × angle |
| 5 | shoulder_lift | 肩关节俯仰 | 0° ~ 180° | 125 ~ 875 | pulse = 500 + 4.1667 × (90° - angle) |
| 4 | elbow | 肘关节 | -90° ~ 90° | 125 ~ 875 | pulse = 500 + 4.1667 × angle |
| 3 | wrist_flex | 腕部俯仰 | -90° ~ 90° | 125 ~ 875 | pulse = 500 + 4.1667 × angle |
| 2 | wrist_roll | 腕部旋转 | -90° ~ 90° | 125 ~ 875 | pulse = 500 + 4.1667 × angle |
| 1 | grip_left | 夹爪开合 | 0° ~ 90° | 200 ~ 700 | pulse = 700 - 5.5556 × angle |

脉宽换算系数 4.1667 = 1000/240，将 240° 的舵机行程映射到 0~1000 的脉宽范围。中心脉宽 500 对应关节 0° 位置。肩关节（舵机5）的换算公式中 angle = 90° - joint_angle，这是因为 STM32 固件中 theta2servo() 函数对肩关节做了翻转处理。

**（3）舵机角度回读帧（CMD_ANGLE_BACK_READING = 13）**

系统在每次关节运动完成后，通过 CMD_ANGLE_BACK_READING 命令回读 6 个舵机的当前脉宽值，与上位机下发指令进行对比验证，实现闭环校验。

- 请求帧：`55 55 02 0D`（共 4 字节）
- 回复帧：22 字节，包含 6 组 (舵机ID, 脉宽低字节, 脉宽高字节)
- 脉宽偏差容忍度：±30（约 ±7°），超出则发出警告

#### 2.5.3 arm_base — 串口通信桥接节点

arm_base 是上位机软件的核心通信节点，使用 C++ 编写，负责串口管理、协议编解码、指令分发以及舵机状态回读验证。该节点是上位机 ROS2 系统与 STM32 下位机之间的唯一数据通道。

**节点架构：**

arm_base 支持两种工作模式，通过 `ik_mode` 参数切换：

- **传统增量坐标模式**（ik_mode=false）：订阅 `/arm_cmd`（增量坐标）和 `/gripper_cmd`（夹爪控制），将坐标增量打包为 CMD_COORDINATE_SET 帧，将夹爪命令打包为 CMD_MULT_SERVO_MOVE 单舵机帧。适用于与 STM32 闭源 IK 库配合使用的场景。

- **IK 模式**（ik_mode=true）：仅订阅 `/joint_target`（6 路关节角度）和 `/grip_cmd`（夹爪独立控制）。关节角度由上位机 ik_node 求解后下发，arm_base 负责角度→脉宽转换和 CMD_MULT_SERVO_MOVE 帧打包发送。该模式完全绕过了 STM32 端的闭源运动学库，上位机对运动过程拥有完全控制权。

两种模式下，arm_base 均发布 `/arm_status`（当前末端位置），供其他节点订阅。

[arm_base 节点内部数据流图 — 展示两种模式的订阅/发布关系]

图3. arm_base 节点数据流图

**关键设计要点：**

**（1）启动握手与重试机制**

arm_base 启动后首先执行版本查询握手：发送 `55 55 02 01` 帧，等待下位机回复 6 字节版本信息。握手成功确认串口通信正常后，才开始接收并处理运动指令。考虑到 STM32 上电启动时间不确定，握手失败时自动重试最多 5 次，每次间隔 500ms。

**（2）STM32 复位时序控制**

Linux 打开串口时 DTR 和 RTS 信号线默认被拉高。本系统中 DTR 连接 STM32 的 RESET 引脚，RTS 连接 BOOT0 引脚。为避免 STM32 在上电时误入 ROM bootloader 模式，arm_base 在打开串口后先拉低 RTS（BOOT0=0），延时 100ms 后再拉低 DTR（释放复位），最后等待 2 秒让 STM32 完成启动。

**（3）主机侧平滑插值**

STM32 固件 `robot_arm_knot_run()` 函数忽略了串口帧中的 `running_time` 参数，硬编码运动时间为 20ms。这意味着如果上位机一次发送大范围关节运动指令，下位机会在 20ms 内完成所有舵机的位置跳变，导致机械臂产生剧烈冲击。

为解决此问题，arm_base 在 IK 模式下实现了主机侧平滑插值算法：将一次大范围运动拆分为多个 20ms 微步，每步发送增量舵机脉宽帧，使舵机平滑过渡到目标位置。插值步数 = 目标运动时间 / 20ms。例如 1500ms 的运动被拆分为 75 个 20ms 微步，避免冲击并实现近似匀速运动。

[插值示意图 — 展示舵机脉宽随时间变化的多步阶梯曲线]

图4. 主机侧平滑插值示意图

**（4）舵机回读与闭环验证**

每次关节运动完成后（含插值完成），arm_base 自动发送 CMD_ANGLE_BACK_READING 命令回读 6 个舵机当前脉宽，与下发指令进行偏差对比。偏差在 ±30（约 ±7°）以内视为正常，超出范围则输出警告日志。该机制可及时发现通信丢帧或舵机堵转等异常情况。

**（5）夹爪独立控制通道**

夹爪（舵机1）通过独立的 `/grip_cmd` topic 控制，不受 `/joint_target` 影响。arm_base 内部维护 `grip_duty_` 状态变量，在插值和关节运动过程中始终锁定夹爪脉宽不变。这避免了 ik_node 发布的 JointState 中 grip_left 字段意外覆盖 task 节点设置的夹爪状态，确保夹爪只在任务节点明确指令下才动作。

**节点参数列表：**

| 参数名 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| usart_port_name | string | /dev/ttyUSB0 | 串口设备路径 |
| serial_baud_rate | int | 9600 | 串口波特率 |
| ik_mode | bool | false | 是否启用 IK 模式 |
| move_time_ms | int | 1500 | 默认运动时间（ms） |
| gripper_servo_id | int | 1 | 夹爪舵机 ID |

#### 2.5.4 ik_node — 逆运动学求解节点

ik_node 是上位机逆运动学的核心实现，使用 Python 编写。该节点订阅 `/arm_cmd`（绝对目标坐标 x, y, z，单位 cm），通过解析几何法求解 6 个关节的目标角度，发布 `/joint_target`（sensor_msgs/JointState）供 arm_base 下发执行。同时发布 `/arm_status` 供标定采集等节点使用。

**运动学建模：**

机械臂为 4-DOF 串联结构（底座旋转 + 3 连杆）加腕部旋转和夹爪，共 6 个关节。DH 参数如下（来源于 STM32 固件 `kinematics.h`）：

| 参数 | 数值 (cm) | 说明 |
|------|-----------|------|
| L1 | 2.89 | 底座垂直偏移（肩部高度） |
| L2 | 10.43 | 上臂长度 |
| L3 | 8.90 | 前臂长度 |
| L4 | 17.70 | 腕部至夹爪末端长度 |

**逆解算法流程：**

逆运动学求解采用解析几何法，将 6-DOF 问题分解为若干子问题逐步求解：

1. **底座旋转角 θ₁**：由目标点在水平面的投影直接计算，θ₁ = atan2(y, x)。

2. **俯仰角 α 自动选取**：由于 4-DOF 平面机构解算 3-DOF 目标位置存在 1 个冗余自由度（末端俯仰角 α），系统自动扫描 α ∈ [-90°, 0°]（步长 1°），筛选满足关节限位的有效俯仰角。默认优先选取 -50°（自然姿态），若不可达则选取最接近的有效值。用户可通过 `preferred_pitch` 参数调整偏好，如设为 -90° 可实现顶部垂直抓取。

3. **腕部位置解算**：根据俯仰角 α 计算腕部在 (r, z) 平面内的位置：
   - r_wrist = r - L4 × cos(α)
   - h_wrist = z - L1 - L4 × sin(α)

4. **两连杆 IK 求解**：在 (r_wrist, h_wrist) 平面内，利用余弦定理求解上臂（L2）和前臂（L3）的二连杆逆运动学，得到两组解（肘部朝上和肘部朝下）。优先选取肘部接近 -45° 的解，使关节远离限位边界。

5. **腕部俯仰角**：θ₄ = α - θ₂ - θ₃。

6. **腕部旋转角**：θ₅ = 0（默认保持中立，可由任务节点覆盖）。

[逆运动学几何示意图 — 展示 (r,z) 平面内的连杆几何关系]

图6. 逆运动学几何求解示意图

求解过程中对每个关节角度进行限位检查，超出范围则排除该解：

| 关节 | 最小值 | 最大值 |
|------|--------|--------|
| shoulder_pan (底座) | -90° | 90° |
| shoulder_lift (肩) | 0° | 180° |
| elbow (肘) | -90° | 90° |
| wrist_flex (腕俯仰) | -90° | 90° |

|-----|-------------|-------------|

#### 2.5.5 arm_cmd — 坐标指令节点

arm_cmd 是坐标指令的入口节点，使用 C++ 编写。在传统模式下，该节点接收目标绝对坐标后，通过订阅 `/arm_status` 获取当前末端位置，计算增量 (dx, dy, dz) 后发布 `/arm_cmd` 增量指令。在 IK 模式下，该节点直接发布目标绝对坐标，由 ik_node 完成逆解后通过 `/joint_target` 下发。

#### 2.5.6 arm_task — 茶艺任务调度节点

tea_task_node 是应用层的任务编排节点，使用 Python 编写。该节点实现了完整的茶艺制作流程，将"目标检测 → 运动规划 → 夹取控制 → 倾倒动作"串联为一个自动化状态机。

**任务状态机设计：**

每个任务步骤遵循严格的三阶段执行模式：

```
MOVING → ARRIVED → SETTLING → ACTING → MOVING → ...
```

1. **MOVING**：发布目标绝对坐标 `/arm_cmd`，由 ik_node 逆解后驱动机械臂运动
2. **ARRIVED**：等待 arm_base 插值完成（等待 move_time_ms）
3. **SETTLING**：静止保持 2.0 秒，让机械臂稳定到达目标位置
4. **ACTING**：执行夹爪开合、腕部翻转等末端动作

**茶盒（caddy）任务流程：**

1. 移动到茶盒上方 → 2. 下降至抓取高度 → 3. 闭合夹爪 → 4. 抬起 → 5. 移动到茶杯上方 → 6. 倾倒（腕部旋转 -90°）→ 7. 返回茶盒位置 → 8. 下降 → 9. 张开夹爪 → 10. 抬起离开

**茶壶（bottle/teapot）任务流程：**

1. 张开夹爪回到初始位置 → 2. 移动到茶壶上方（Y轴补偿-3cm）→ 3. 下降至抓取高度 → 4. 闭合夹爪 → 5. 回撤至 x=15 安全位置并抬高 → 6. 移动到茶杯前方倒茶位置 → 7. 倾倒（腕部俯仰 -60°，模仿人手倒茶）→ 8. 回撤安全位置 → 9. 返回茶壶上方 → 10. 下降 → 11. 张开夹爪释放 → 12. 抬起 → 13. 回到初始位置

**安全机制：**

- **位置冻结**：任务执行期间，YOLO 检测结果更新被锁定，避免目标位置跳变导致机械臂失控
- **安全通过高度**：在目标物体上方设置过渡高度（transit_z），机械臂先在安全高度水平移动，再垂直下降抓取，避免与工作台面碰撞
- **最大可达高度校验**：根据 XY 坐标实时计算该位置的最大可达 Z 值，确保目标位置在机械臂工作空间内
- **夹爪独立控制**：夹爪通过独立的 `/grip_cmd` topic 控制，与手臂关节运动通道隔离，避免相互干扰

**夹爪开合参数：**

| 状态 | 角度 | 说明 |
|------|------|------|
| 完全张开 | 90° (π/2 rad) | 准备抓取 |
| 抓取茶盒 | 25° | 茶盒较宽，不需要完全闭合 |
| 抓取茶壶 | 15° | 茶壶手柄较细，需要更紧的夹持力 |

#### 2.5.7 启动配置与系统集成

系统提供两个 launch 文件用于不同场景的快速启动：

**（1）ik_bringup.launch.py — IK 调试启动**

用于逆运动学调试和单点运动测试，同时启动 arm_base（IK 模式）、ik_node 和 arm_cmd：

```
ros2 launch arm_test ik_bringup.launch.py target_x:=15.0 target_y:=0.0 target_z:=2.0
```

**（2）tea_task.launch.py — 茶艺任务启动**

用于完整茶艺任务执行，同时启动 arm_base（IK 模式）、ik_node 和 tea_task_node：

```
ros2 launch arm_task tea_task.launch.py move_time_ms:=1500
```

任务触发方式：
```
# 执行茶盒任务
ros2 topic pub -1 /task_cmd std_msgs/msg/String "data: '1'"

# 执行茶壶任务
ros2 topic pub -1 /task_cmd std_msgs/msg/String "data: '2'"
```

**完整系统数据流：**

```
视觉检测 (YOLOv8)
    │ /target_grasp_array (目标类别+3D坐标)
    ▼
手眼标定矩阵 T_base_camera
    │ 坐标转换: P_base = R × P_camera + t
    ▼
tea_task_node (任务调度)
    │ /arm_cmd (绝对目标坐标 x,y,z) + /grip_cmd (夹爪角度)
    ▼
ik_node (逆运动学求解)
    │ /joint_target (6路关节角度) + /arm_status (当前末端位置)
    ▼
arm_base (串口通信桥接)
    │ CMD_MULT_SERVO_MOVE / CMD_COORDINATE_SET (二进制帧)
    ▼
STM32 下位机
    │ USART2 → 总线舵机协议
    ▼
6 路舵机执行
```

## 第三部分 完成情况及性能参数

### 3.2.3 软件成果（机械臂上位机部分）

软件部分已形成完整的 ROS2 节点链路。arm_base 实现了串口通信管理、协议编解码、主机侧平滑插值和舵机状态闭环回读；ik_node 实现了基于解析几何的 6-DOF 逆运动学求解，支持俯仰角自动选优和工作空间约束；arm_cmd 实现了增量/绝对坐标双模式指令下发；tea_task_node 实现了茶盒抓取倾倒和茶壶抓取倾倒两条完整任务流水线。

核心数据通过 `/arm_cmd`（目标坐标）、`/joint_target`（关节角度）、`/arm_status`（末端位置）、`/grip_cmd`（夹爪控制）和 `/task_cmd`（任务触发）等 Topic 传递。各节点职责单一，通过 Topic 解耦，便于单独定位问题和替换模块。
